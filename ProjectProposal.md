<h1>프로젝트 기획의 배경</h1>
1. 교통사고 후유증으로 즉시 기억의 반환의 속도가 상당히 더딘 '개인적인 이유'와 목적에서 개발을 시작함.
2. 해당 부분에 대해서 작가로 활동하며, AI가 나왔고 그 AI를 활용한 개발의 안전한 소설의 관리와 도움이 가능하리라는 가능성을 발견함.
3. 또한, 내 단점을 해결하기 위한 최대한의 가능성을 발견함.

<h1>프로젝트 기획의 이유</h1>

1. 소설 작성 시 일정 관리의 불편.
2. 소설 설정이나 장면의 빠른 서치 필요.
3. 작업 량의 수월한 측정.
4. 작업 시간의 수월한 측정.
5. AI를 통한 안정적인 작업 서포트.
6. 체계적으로 그동안 작업한 소설에 접근하여 필요 시 참고.

<h1>각각의 필요한 프로젝트 작업 개요</h1>

1. TreeView에서 문서 탭을 조작하여 문서를 드래그 앤 드랍으로 순서 변경 및 조작할 수 있도록 설정.
2. Doucments의 Form을 설정하도록 작업.
3. Project 생성, Project에 Group(소설)과 Page(내용), Information(설정) 등을 생성할 수 있게 생성.
4. HWPX 파일 및 HWP 파일을 읽고 쓰도록 설정.
5. WikiTree 또는 NamuWiki 등으로 설정을 작업할 수 있도록 형성.
6. 작업 시간 측정을 위한 Timer 포함.
7. Timer를 통한 작업 누적율과 평균 작업율 등을 알기 쉽게 정리.
8. AI ChatView를 통한 작업 도움 및 기존 작업 브리핑을 위한 기능 구현.
9. AI ChatView를 접속시 그날 작업 분량의 리뷰를 설정.

<h1>향후 기능</h1>
1. 우선 Ollama를 통한 데이터 처리를 하지만, llama.cpp를 통한 로컬 AI의 호출을 통해 편의성을 증대함.
2. UI 커스텀 기능을 제공.
3. 정보 접근 제외 기능을 제공.
4. Google Drive 등을 통한 백업 지원.


<h2>서론</h2>

AI를 소설 기획 또는 경험적 학습에 이용하는 와중에, '개인적인 어려움'을 AI를 통해 해결할 수 있었으면 좋겠다는 생각이 떠올랐다.
정확하게는 본디 AI가 아니라 머신 러닝 알고리즘을 활용하여, AI가 아닌 시간 관리의 정향성을 추가하는 편이 좋겠다는 생각을 했다.
원래는 Timer를 MacOSX를 기반으로 개발하던 도중에, 고사양의 윈도우 PC를 새로 맞추게 되면서 내가 원하는 본질적인 도움을 위한 소프트웨어 개발을 시작함.
그 프로그램은 어디까지나, AI를 외장 메모리로 하여 내 취약점을 보완하겠다는 생각에서 시작한 프로젝트이며, 동시에 AI가 내 생각을 보조해 주고 도움을 준다는 의미에서 EMSW, 즉 Echo Mind Sub Writer 라는 이름으로 정의했다.

개발을 진행하면서, 앞으로 수많은 어려움에 부딪히게 될 것이다.
특히, 기존의 C언어 결합성을 위한 개발에서 분명히 큰 어려움은 가지고 있을 것이다.
왜냐하면, 궁극적으로 Ollama API가 아닌 llama.cpp를 통한 경합성을 목적으로 하고 있기 때문에 더욱 그렇다.

개인적인 필요성으로 개발한 프로그램이 작가 내지는 그것을 넘어 다른 사람들에게 큰 도움이 되었으면 좋겠다는 소망 또한 가지고 있다.

기술적 개발에 대한 문제점과 개발 완수 시간에 대한 고민

첫번째로, Ollama를 기존에 사용하지만, 향후 llama.cpp를 위한 개발 시간은 프로젝트 완성 제한 시간(11월 말 내지는 12월 말)은 명백한 도전이었고, 과제였다.
두번째로, 개인적인 취미를 보다 더 타이트한 개발을 위해 향후 목표를 세우고 바쁜 시간동안 해나가는 것은 더욱 힘든 과정이었다.
세번째로, C와 Python의 결합을 어떻게 해야 하는지 고민을 했으며, 결국 데이터 관리를 위한 것을 목적으로 하고 있음이 더욱 명백해졌다.
네번째로, DataTools.c의 경우, C와 Python간의 메모리 큐를 관리하기 위함과 동시에 Documents를 관리하기 위해서 만들었으나, 추후 llama.cpp와의 연동을 위해서는 Documents와 WikiTree나 NamuWiki 식으로 설정을 정리하는 기능을 제공하기 위해선 코드의 분리가 필수라는 것을 정리했다.
다섯번째로, DataTools.c는 위 의도로 사실상 현 단계에서 필요는 없지만, 추후 기능을 위하여 공개한다.

<h3>난관 예상</h3>

일정의 타이트와는 관계 없이, 기존의 Text를 AI로 처리하면서 데이터의 정제를 어떻게 할지, 메시지 큐를 어떻게 관리해야 AI에 연결이 편리할지. 이것은 결과적으로 중요한 과제이다.

<h2>본론</h2>

우선, 기본적으로 이 프로젝트는 프로젝트를 지원하는 하나의 보조 프로그램으로 관리하며, 사용자의 환경을 체계적으로 지원하기 위해 UI에 대해서는 다음과 같은 목표를 가지고 있다.

1. UI는 별도의 창으로 분리되어 관리한다.
2. 별도의 창은 별도의 Documents.c를 통해 관리한다.
3. Keyboard의 error tesk는 .c 파일을 통해 데이터 관리 단계에서 기본적으로 Keyboard event로 분리한다.
4. keyboard event는 Documents.c에서 처리된다. 단, 창이 In 상태일 경우 Python에서 분리되어 C에서 독립적으로 처리된다.
5. GIL 상태를 관리하기 위하여, Ollama를 연동하여 처리될 경우 별도의 로컬에서 처리되는 것이므로 큰 문제가 없으나, llama.cpp에서 처리되는 경우, 마찬가지로 별도의 ChatQueue를 C로 구현하여 llama.cpp와 연동하여 처리한다.

위와 같은 방식으로 C를 통한 최적화와 기능적 안정성을 제공하며, GIL의 영향을 최대한 줄이는 것으로 활용한다.
또한, 프로젝트는 기본적으로 별도의 파일로 관리하나, 추후 윈도우의 파일 관리 시스템을 지원하기 위하여 ODT의 형태로 데이터를 관리하여, 검색과 색인의 용이성을 높일 것이다.
색인과 검색의 경우 텐서 검색을 통해 AI가 해당 프로젝트를 스켄하여, 스켄한 프로젝트에서 내용을 정확하게 검색하여 해당 데이터를 통한 의견 제공을 돕는다.
